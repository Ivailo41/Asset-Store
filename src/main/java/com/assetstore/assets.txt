Yes, you can use FileChannel to stream a file from a client to an NIO server, but you need to handle it carefully because FileChannel works with files on the local filesystem, not directly with network sockets. With NIO, the process is about bridging the file and the network channel. Here’s the breakdown:

1. Client Side: Streaming File Without Loading Into Memory

You can open a FileChannel for your file and then transfer its contents to a SocketChannel connected to your server:

try (SocketChannel socketChannel = SocketChannel.open(new InetSocketAddress("localhost", 9000));
     FileChannel fileChannel = FileChannel.open(Paths.get("path/to/file"), StandardOpenOption.READ)) {

    ByteBuffer buffer = ByteBuffer.allocateDirect(8192); // Direct buffer avoids copying between JVM heap and OS

    while (fileChannel.read(buffer) > 0) {
        buffer.flip();                   // Prepare buffer for writing
        while (buffer.hasRemaining()) {
            socketChannel.write(buffer); // Write to network
        }
        buffer.clear();                  // Prepare buffer for next read
    }
}

Notes:

allocateDirect() is recommended for large files to reduce JVM overhead.
This approach never loads the full file into memory, just chunks in the buffer.
2. Server Side: Receiving File Using NIO

On the server, you can use a FileChannel to write directly to disk as you read from the SocketChannel:

try (ServerSocketChannel serverChannel = ServerSocketChannel.open()) {
    serverChannel.bind(new InetSocketAddress(9000));
    SocketChannel client = serverChannel.accept();

    try (FileChannel fileChannel = FileChannel.open(Paths.get("path/to/save/file"),
            StandardOpenOption.CREATE, StandardOpenOption.WRITE)) {

        ByteBuffer buffer = ByteBuffer.allocateDirect(8192);

        while (client.read(buffer) > 0) {
            buffer.flip();
            while (buffer.hasRemaining()) {
                fileChannel.write(buffer); // Stream directly to disk
            }
            buffer.clear();
        }
    }
}

Key Points:

The server doesn’t need to buffer the whole file in memory, it writes as chunks arrive.
Using ByteBuffer.allocateDirect() can improve performance for large file streams.
3. Optional Optimization: FileChannel.transferTo / transferFrom

On local-to-local transfer, FileChannel has transferTo and transferFrom which can use OS-level zero-copy. However, they only work between FileChannel and WritableByteChannel, so if you want SocketChannel → FileChannel, transferFrom can work:

fileChannel.transferFrom(socketChannel, 0, Long.MAX_VALUE);

Caveats:

transferFrom may block depending on network behavior.
You usually still need to handle partial reads if the channel is non-blocking.
✅ Conclusion
Yes, FileChannel is suitable for streaming files in NIO without loading into memory.
Use direct ByteBuffers for efficiency.
On the server, write chunks as they arrive to disk.
Optional: transferTo/transferFrom can simplify code for blocking channels.

when does transferFrom block

FileChannel.transferFrom(ReadableByteChannel src, long position, long count) can block depending on the type of ReadableByteChannel you pass and the mode of operation. Let’s break it down carefully:

1. Blocking vs Non-blocking Channels
Blocking channels (default for SocketChannel)
transferFrom will block until data is available to read from src or until EOF is reached.
Example: A normal TCP SocketChannel in blocking mode → the method waits for bytes from the client.
Non-blocking channels (SocketChannel.configureBlocking(false))
transferFrom may return 0 immediately if no data is available, because non-blocking channels never wait.
You cannot rely on it to transfer all bytes at once in non-blocking mode; you need a loop with a selector.
2. When transferFrom blocks

It blocks when:

The src channel is in blocking mode.
The operating system cannot immediately transfer the requested number of bytes (maybe due to TCP flow control or disk I/O).

It does not guarantee it will transfer all requested bytes (count). It returns the number of bytes actually transferred. So even in blocking mode, you might need:

long transferred = 0;
while (transferred < fileSize) {
    transferred += fileChannel.transferFrom(socketChannel, transferred, fileSize - transferred);
}
3. Non-blocking usage

If your SocketChannel is non-blocking:

transferFrom might return 0 if no data is ready.
You must use a Selector to wait for OP_READ before calling transferFrom again.
✅ Rule of Thumb
transferFrom blocks only if the source channel is in blocking mode and data is not immediately available.
It may transfer fewer bytes than requested, so always check the return value and loop until done.

So if I send the server a header with the file size I can just use the FileChannel to transfer the file to the server disk

Exactly — that’s a solid approach, and it’s actually a common pattern for file streaming with NIO. Let me lay it out step by step so it’s clear and safe:

1. Send a Header First
Before sending the file data, send a small header over the socket.
The header contains essential info like:
File size (long is enough for very large files)
File name (optional)
Example: you can use a fixed-size 8-byte long for file size.
ByteBuffer header = ByteBuffer.allocate(Long.BYTES);
header.putLong(fileSize);
header.flip();
socketChannel.write(header);
2. Server Reads the Header
Read the fixed-size header first to know how many bytes to expect.
This allows the server to know when the transfer is complete.
ByteBuffer header = ByteBuffer.allocate(Long.BYTES);
while(header.hasRemaining()) {
    client.read(header);  // For non-blocking, use a selector and check OP_READ
}
header.flip();
long fileSize = header.getLong();
3. Use FileChannel.transferFrom
Once you know the file size, you can use transferFrom to stream bytes directly from the SocketChannel to disk:
try (FileChannel fileChannel = FileChannel.open(Paths.get("received_file"),
        StandardOpenOption.CREATE, StandardOpenOption.WRITE)) {

    long position = 0;
    while (position < fileSize) {
        long transferred = fileChannel.transferFrom(client, position, fileSize - position);
        if (transferred == 0) {
            // Non-blocking mode: wait until data is available
            Thread.yield(); // or use selector for OP_READ
        }
        position += transferred;
    }
}

Notes:

This approach never loads the whole file into memory. Only a small OS-level buffer is used internally.
Works for very large files as long as the disk has space.
Using a header ensures you know exactly how many bytes to read, avoiding premature EOF or leftover bytes.
4. Optional Enhancements
Include file metadata in the header (e.g., file name, checksum).
If using non-blocking SocketChannel, integrate with a Selector to avoid busy-waiting.
